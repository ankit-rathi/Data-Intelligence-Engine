# **From Data to Decisions**

## Building Economic Accountability for Data Projects in Banking

---

# COURSE ARCHITECTURE

**Flow:**
Problem → Economic Lens → Structured Framework → Governance Mechanism → Real Case Application

Not academic.
Not tool-driven.
Decision-driven.

---

# MODULE 1 — The Problem: Why We Struggle to Prove Value

### Objective:

Create discomfort and urgency.

### Content:

* We can quantify project cost.
* We cannot quantify business impact.
* Post-launch value discussions are vague.
* “Feels useful” ≠ defensible ROI.
* This is systemic across data teams.

### Case Reference:

Briefly introduce:

* Dashboard project (couldn’t quantify uplift initially)
* DSAR project (clearly risk-driven but not framed economically)

Key Insight:
Data success ≠ economic success.

---

# MODULE 2 — First Principles: What Is a Decision?

### Objective:

Re-anchor thinking.

Core ideas:

* A decision = choice under uncertainty.
* Every decision commits capital.
* Business value = revenue, cost, or risk movement.
* No decision change → no value.

Introduce the decision chain:

Data → Insight → Decision → Action → Economic Outcome

Breaks usually happen between insight and economic outcome.

---

# MODULE 3 — Expected Value Thinking for Banking

### Objective:

Introduce probabilistic economics without math overload.

Concepts:

* Expected Value = Probability × Impact
* Baseline vs Counterfactual
* Frequency matters
* Risk is probabilistic, not emotional

Apply lightly to:

* Dashboard (optimization frequency quarterly)
* DSAR (12,000+ compliance events annually)

Key shift:
We don’t measure outcomes.
We estimate uplift probabilistically.

---

# MODULE 4 — Mapping Data Projects to Economic Levers

### Objective:

Force classification discipline.

Every project must map to:

1. Revenue Expansion
2. Cost Reduction
3. Risk Mitigation

Deep dive examples:

### Case Study 1 — Strategic Dashboard

* Lever: Cost + Risk Integrity
* Frequency: Quarterly
* Economic Pathways:

  * Manual cost reduction
  * Improved redundancy detection
  * Reduced reporting error probability

### Case Study 2 — DSAR Automation

* Lever: Risk + Cost
* Frequency: 30–40 daily events
* Economic Pathways:

  * Reduced SLA breach probability
  * Lower regulatory exposure
  * Operational cost savings
  * Scalability insurance

Participants see contrast between:
Soft-value ambiguity vs Hard-risk quantifiability.

---

# MODULE 5 — The Decision-Value Operating System (The Framework)

Now introduce the structured method.

## The 6-Step Decision Impact Loop:

1. Define the Decision Clearly
2. Quantify Baseline
3. Identify Economic Lever
4. Estimate Uplift per Decision
5. Multiply by Frequency
6. Compare Against Project Cost

No jargon.
Just structure.

Apply both case studies step-by-step.

This is where clarity becomes practical.

---

# MODULE 6 — Risk Quantification in Regulated Environments

### Objective:

Teach how to quantify risk when outcomes are uncertain.

Topics:

* Probability of breach
* Severity of impact
* Expected regulatory exposure
* Reputational spillover
* Audit defensibility value

Deep Dive:
Reconstruct DSAR expected risk model.

This is powerful for banking audience.

---

# MODULE 7 — Why Most Data Projects Fail at Attribution

Common structural failures:

* Decision not defined upfront
* Baseline not measured
* No counterfactual design
* No instrumentation for post-launch review
* Solution built before economic case

Use dashboard case to illustrate.

This creates institutional reflection.

---

# MODULE 8 — The Decision Impact Charter

Now introduce the governance mechanism.

Before funding, every project must answer:

1. What decision is improved?
2. Who makes it?
3. How often?
4. What is current baseline performance?
5. What is economic exposure?
6. What is expected annual value?
7. What is solution cost?
8. What are invalidation triggers?

Participants draft a charter for:

* One past project
* One current live project

This makes it applied, not theoretical.
